## 0. TL;DR

读完美团技术服务文档（`/Users/reedom/PyCharmMiscProject/美团到店餐饮技术服务.md`）后发现 5/12 brainstorm 的「Path A 轮询日报」**架构前提不成立**：美团只提供 webhook 消息推送，没有「营业日报 API」。

新方案：**webhook 实时接收 + append-only 原始消息表 + 单笔 RevenueTransaction 入账 + incremental upsert 日聚合 RevenueRecord**。

接入全 6 种 msgType：
- **110009 代金券买单**（必接）→ 写 RevenueTransaction(channel=MEITUAN_VOUCHER)
- **110029 新订单已支付** → 写 RevenueTransaction(channel=MEITUAN_ORDER) + 同步返 vendorOrderId
- 110011/110019/110021/110027 → 轻量处理（状态字段更新 + OpLog + 通知）

**双层防御**：
1. MeituanRawMessage append-only 表保留所有原始 payload，业务失败可重放
2. 同步问 BD 是否有「补拉 API」当兜底（可选）

## 3. 数据模型

### 3.1 新增表 `MeituanRawMessage`（原始消息 append-only）

```prisma
model MeituanRawMessage {
  id              String        @id @default(cuid())
  msgId           String        @unique           // 美团 msgId，幂等防重放
  msgType         Int                              // 110009 / 110011 / 110019 / 110021 / 110027 / 110029
  developerId     Int
  businessId      Int
  ePoiId          String?                          // 服务商门店 id
  opBizCode       String?
  msgTimestamp    BigInt                           // 美团推送 timestamp（秒）
  rawHeaders      Json                             // 完整 HTTP headers
  rawBody         String        @db.Text           // form-urlencoded 原文
  rawMessage      Json                             // message 字段解析后的 JSON

  status          MeituanMsgStatus @default(RECEIVED)
  receivedAt      DateTime      @default(now())
  processedAt     DateTime?
  failReason      String?
  retryCount      Int           @default(0)

  resultTxId      String?                          // 关联 RevenueTransaction.id（如有产出）

  @@index([msgType, status])
  @@index([receivedAt(sort: Desc)])
  @@map("meituan_raw_messages")
}

enum MeituanMsgStatus {
  RECEIVED        // 收到 + 验签通过 + 落库
  PROCESSED       // 业务处理成功
  FAILED          // 业务处理失败（可重放）
  IGNORED         // 非营收 msgType 或未知 msgType，仅记录不处理
}
```

### 3.2 新增表 `RevenueTransaction`（单笔流水）

```prisma
model RevenueTransaction {
  id              String        @id @default(cuid())
  tenantId        String
  storeId         String
  channel         RevenueChannel
  externalOrderNo String                           // 美团 orderId 字符串
  externalMsgId   String?                          // 关联 MeituanRawMessage.msgId

  amount          Decimal       @db.Decimal(12, 2) // 客户实付金额（元）
  refundAmount    Decimal       @default(0) @db.Decimal(12, 2)
  netAmount       Decimal       @db.Decimal(12, 2) // amount - refundAmount

  paidAt          DateTime                         // 平台口径支付时间
  refundedAt      DateTime?
  status          RevenueTxStatus @default(NORMAL)

  rawPayload      Json                             // 落库时的完整业务数据
  reconciledAt    DateTime?                        // 对账完成时间（Path C 才用）

  createdAt       DateTime      @default(now())
  updatedAt       DateTime      @updatedAt

  tenant          Tenant        @relation(fields: [tenantId], references: [id])
  store           Store         @relation(fields: [storeId], references: [id])

  @@unique([channel, externalOrderNo])             // 防同笔订单重复入账
  @@index([storeId, paidAt(sort: Desc)])
  @@index([status])
  @@map("revenue_transactions")
}

enum RevenueChannel {
  MEITUAN_VOUCHER       // 110009 代金券买单
  MEITUAN_ORDER         // 110029 新订单已支付（MOP + 团购）
  DOUYIN_VOUCHER        // 抖音代金券（预留）
  DOUYIN_ORDER          // 抖音订单（预留）
  AGGREGATOR            // 聚合支付桌签码（预留）
  WECHAT_MINI           // 微信小程序（预留）
  ALIPAY_MINI           // 支付宝小程序（预留）
}

enum RevenueTxStatus {
  NORMAL                // 正常营收
  REFUNDED              // 已全退（refundAmount = amount）
  PARTIAL_REFUND        // 部分退款
  CANCELED              // 订单取消
  ORPHAN                // 本地有 / 平台无（异常）
}
```

### 3.4 `RevenueRecord` 不动 — incremental upsert 由 webhook handler 同步触发

每笔 RevenueTransaction 入库时，**同一个事务里**对 RevenueRecord 做 incremental upsert：

```ts
await prisma.$transaction([
  prisma.revenueTransaction.create({ data: txData }),
  prisma.revenueRecord.upsert({
    where: { storeId_date: { storeId, date: paidAtDate } },
    create: { storeId, date: paidAtDate, amount: tx.netAmount, source: 'meituan' },
    update: { amount: { increment: tx.netAmount } },

```

退款时同样用 `{ decrement: refundAmount }` 原子操作。前端 dashboard / 报表读 RevenueRecord（聚合视图，性能更好），明细路径读 RevenueTransaction。

### 4.3 签名模块（可插拔）

```ts
export interface SignatureVerifier {
  verify(params: Record<string, string>, expectedSign: string): boolean
}

/**
 * 默认实现：业内通用美团 webhook 签名
 * - 取除 sign 外所有参数
 * - 按 key 字典序排序
 * - 拼接 key1=val1&key2=val2&...
 * - 末尾追加 appSecret
 * - SHA1 → 40 字符 hex
 *
 * ⚠️ TODO: 待 BD 提供权威文档后校准
 */
export class StandardMeituanSignature implements SignatureVerifier {
  constructor(private appSecret: string) {}

  verify(params: Record<string, string>, expectedSign: string): boolean {
    const sorted = Object.keys(params)
      .filter(k => k !== 'sign')
      .sort()
      .map(k => `${k}=${params[k]}`)
      .join('&')

    const computed = crypto.createHash('sha1')
      .update(sorted + this.appSecret).digest('hex')

    return crypto.timingSafeEqual(
      Buffer.from(computed, 'hex'),
      Buffer.from(expectedSign, 'hex'),
    )
  }
}

export function createVerifier(): SignatureVerifier {
  return new StandardMeituanSignature(process.env.MEITUAN_APP_SECRET!)
}
```

实际签名规则跟业内标准不一致时，**只改 signature.ts 一个文件**，不影响其它代码。

### 4.4 主处理流（伪代码）

```ts
app.post('/api/webhooks/meituan', {
  config: { skipIdempotency: true, skipRateLimit: true },
}, async (req, reply) => {
  const rawBody = req.body as string
  const params = parseFormUrlencoded(rawBody)

  // 1. 验签（失败前不解 message JSON，防大 payload 攻击）
  if (!verifier.verify(params, params.sign)) {
    return reply.code(401).send({ code: -1, message: 'signature invalid' })
  }

  // 2. 解 message
  let messageJson: unknown
  try {
    messageJson = JSON.parse(params.message)
  } catch {
    return reply.code(400).send({ code: -2, message: 'message json invalid' })
  }

  // 3. 落 MeituanRawMessage（用 unique 约束做幂等）
  const created = await prisma.meituanRawMessage.create({
    data: {
      msgId: params.msgId,
      msgType: Number(params.msgType),
      developerId: Number(params.developerId),
      businessId: Number(params.businessId),
      ePoiId: params.ePoiId || null,
      opBizCode: params.opBizCode || null,
      msgTimestamp: BigInt(params.timestamp),
      rawHeaders: req.headers,
      rawBody,
      rawMessage: messageJson,
      status: 'RECEIVED',
    },
  }).catch(err => {
    if (err.code === 'P2002') return null   // msgId 重放命中 unique
    throw err
  })

  if (!created) {
    return reply.send({ code: 0, message: 'success' })   // 重放认为成功
  }

  // 4. 110029 同步快路径
  if (created.msgType === 110029) {
    try {
      const result = await processor.processSync(created.id)
      return reply.send({ code: 0, message: 'success', data: result })
    } catch (err) {
      // 同步处理失败也返成功（仅记录），靠 FAILED 重放兜底
      Sentry.captureException(err)
      return reply.send({ code: 0, message: 'success', data: {} })
    }
  }

  // 5. 其他 msgType 异步处理
  reply.send({ code: 0, message: 'success' })
  setImmediate(() => {
    processor.processAsync(created.id).catch(err => {
      req.log.error({ err, msgId: params.msgId }, 'meituan async process fail')
    })
  })
})
```

**关键设计点**：
- 验签前不解 message JSON（防内存攻击）
- 用 DB unique 约束做幂等（比先查后插更原子）
- 110029 走同步快路径（满足返 `data.vendorOrderId` 的可能要求）
- 其他 5 个 msgType 走 `setImmediate` 异步
- 响应永远 `code: 0`，除非验签/格式错；业务失败靠 status=FAILED 后台重放

## 5. 6 个 msgType handler 详解

### 5.1 行为总表

| msgType | 同步/异步 | 主要动作 | 写表 | 通知 |
|---|---|---|---|---|
| **110009 代金券买单**【必接】 | 异步 | 解 msgType 子状态 → 入账或更新退款 | RevenueTransaction | FAILED → ADMIN |
| **110029 新订单已支付** | **同步快路径** | 入账 + 返 vendorOrderId | RevenueTransaction | FAILED → ADMIN |
| 110011 开通代金券买单 | 异步 | UPDATE Store.meituanVoucherEnabled | Store | ADMIN 知晓 |
| 110021 设置变更 | 异步 | OpLog + 推通知（让老板手工去美团后台看） | OpLog | ADMIN |
| 110019 门店授权同步 | 异步 | UPDATE Store.meituanLastAuthAt；找不到店则告警 | Store / OpLog | ADMIN（找不到时 P1） |
| 110027 审核驳回 | 异步 | OpLog + 推 reject reason | OpLog | ADMIN |

### 5.1.1 Store 解析规则（每个 handler 从哪个字段找 Store）

美团各 msgType 的 shopId 字段位置**不统一**，实施时务必按下表对应（来自文档字段对照）：

| msgType | shopId 字段来源 | 备注 |
|---|---|---|
| 110009 代金券买单 | **外层 `params.ePoiId`**（公共参数） | message 体里**没有** shopId 字段 |
| 110029 新订单已支付 | `message.vendorShopId` | message 体里有 |
| 110011 开通代金券买单 | **外层 `params.ePoiId`** | message 体里**没有** |
| 110019 门店授权同步 | `message.vendorShopId` | message 体里有 |
| 110021 设置变更 | `message.poiId`（long 类型，需转 string 匹配） | 注意类型转换 |
| 110027 审核驳回 | `message.vendorShopId`（scope=2 时）；scope=1 时无门店级 | 总部级驳回不绑店 |

统一的解析 helper：

```ts
function resolveStoreFromMessage(raw: MeituanRawMessage): Promise<Store | null> {
  const shopId = (() => {
    switch (raw.msgType) {
      case 110009: case 110011: return raw.ePoiId                       // 外层
      case 110029: case 110019: return raw.rawMessage.vendorShopId       // 内层
      case 110021: return String(raw.rawMessage.poiId)                   // long → string
      case 110027:
        return raw.rawMessage.scope === 2 ? raw.rawMessage.vendorShopId : null
      default: return null
    }
  })()
  if (!shopId) return Promise.resolve(null)
  return prisma.store.findFirst({ where: { meituanShopId: shopId } })
}
```

### 5.5 Handler 公共错误处理

```ts
// processor.ts
async function processMessage(rawMsgId: string) {
  const raw = await prisma.meituanRawMessage.findUnique({ where: { id: rawMsgId } })
  if (!raw || raw.status !== 'RECEIVED') return

  try {
    const handler = HANDLERS[raw.msgType]
    if (!handler) {
      await markStatus(raw.id, 'IGNORED', `unknown msgType ${raw.msgType}`)
      return
    }
    const result = await handler(raw)
    await markStatus(raw.id, 'PROCESSED', null, result?.txId)
  } catch (err) {
    await markStatus(raw.id, 'FAILED', err.message)
    await prisma.opLog.create({ data: { /* 失败 OpLog */ } })
    await notifyAdmin('meituan_handler_failed', { msgId: raw.msgId, msgType: raw.msgType, error: err.message })
    Sentry.captureException(err, { extra: { msgId: raw.msgId, msgType: raw.msgType } })
  }
}
```

**重放机制**：管理员后台点 `重放` → UPDATE status='RECEIVED' → processor 巡检捡起。

### 6.5 监控告警

| 告警条件 | 通知 | 级别 |
|---|---|---|
| 连续 30min 无 webhook（10:00-22:00 营业时段） | ADMIN | P1 |
| FAILED 数 > 5（24h） | ADMIN | P2 |
| 单笔 amount 偏离均值 ±5x | FINANCE | P2 |
| 110019 出现「未匹配 vendorShopId」 | ADMIN | P1 |
| 验签失败 > 10（1h） | ADMIN | P1（疑似攻击） |
| processor 未捕获异常 | Sentry | P1 |

Sentry 仅上报 5xx + processor 异常，4xx 业务错（验签/重放）静默。

### 6.6 Capacitor APP 影响

**零影响**。webhook 是服务器对服务器调用。手机端 app 改不改、apk 发不发都不影响 webhook 链路。老板用浏览器或 V4 app 看 `/v2/boss/integrations/meituan` 即可。

## 9. 关键决策记录

| 决策点 | 选项 | 选了 | 理由 |
|---|---|---|---|
| 接入范围 | 仅必接 vs 营收必需 vs 全 6 个 | **全 6 个** | 用户决策；非营收 msgType 也有业务价值（开通通知 / 授权同步 / 设置变更 / 审核驳回） |
| 110009 vs 110029 是否合并 | 合并去重 vs 双轨独立 | **双轨独立** | 业务上是两套不同订单系统，dashboard 上分两条数据线显示更清晰；用 (channel, externalOrderNo) 唯一约束防真正的重复 |
| 签名实现 | 写死 vs 可插拔 | **可插拔模块** | BD 文档没给规则，业内标准做默认，联调发现差异时只改一个文件 |
| webhook 丢消息兜底 | 接受 vs 双层 vs 三层 | **双层（raw 表 + 找 BD 拉 API）** | 单店少量丢失影响小，但 raw 表必做（重放兜底应用层 bug），补拉 API 当兜底兜底 |
| 110029 同步 vs 异步 | 同步 vs 异步 | **同步快路径** | 文档暗示要返 `data.vendorOrderId`，保守起见同步 |
| RevenueRecord 卷出方式 | 实时 incremental vs 日批 | **实时 incremental upsert** | 同事务原子操作 `{ increment: }` 无竞态，dashboard 当日数据准 |
| 凭证存哪 | env vs DB 表 | **env** | 单店、单租户、不需 UI 加密；多店时再迁 PlatformCredential 表 |
